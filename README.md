## 📌 Кейс: Оптимізація аналітичного дашборду для Big Data датасету Нью-Йоркського таксі

### 1. Situation (Ситуація)

Компанія мала великий масив сирих даних про поїздки міського таксі (понад **40 мільйонів рядків**). Директору був необхідний оперативний дашборд для моніторингу бізнес-метрик (прибутковість, середній чек, чайові, тривалість поїздок) та інтерактивна карта завантаженості міста. Пряме підключення інструменту візуалізації (Power BI) до сирої бази даних призводило до критичного гальмування: через Full Table Scan та важкі гео-координати з GPS-трекерів (точність до міліметра) база «закипала», а звіти не завантажувалися.

### 2. Task (Завдання)

Розробити та оптимізувати архітектуру БД (PostgreSQL) та аналітичний шар для Power BI так, щоб:

1. Очистити дані від системного бруду та GPS-аномалій.
2. Згрупувати мільйони точок у зрозумілі міські «квартали» без втрати бізнес-логіки.
3. Організувати щоденне автооновлення.
4. **Головне:** Забезпечити миттєве (до 1 секунди) перемикання фільтрів та рендеринг карти на дашборді з боку користувача.

### 3. Action (Дії)

Я реалізував комплексний підхід до оптимізації на трьох рівнях (БД, SQL-код, BI):

* **Проектування архітектури:** Налаштував партиціювання таблиць за місяцями для оптимізації зберігання та паралельного зчитування.
* **Data Cleaning (Очищення даних):** Написав скрипти фільтрації для відсікання аномалій: нульові/від'ємні чеки, нульова кількість пасажирів, поїздки-помилки трекера ($>1000$ миль) та географічні викиди поза межами штату Нью-Йорк.
* **Математичне стиснення координат:** Замість використання важких гео-інструментів, я застосував трюк округлення GPS-координат за допомогою `ROUND(latitude/longitude, 3)`. Це дозволило об'єднати міліметрові точки у фіксовану сітку міських кварталів (~110 метрів) прямо в базі.
* **SQL-оптимізація:** Переніс фільтрацію низькочастотних зон у блок `HAVING count(*) > 10` всередині СТЕ. Це відсікло мільйони «глухих» рядків на ранньому етапі й зменшило час виконання запиту з 63 до 46 секунд на сирій таблиці, що дозволило і зменшити навантаження на сервер.
* **Кешування (Materialized View):** Оскільки Power BI імпортував дані без жорстких фільтрів дат у самому SQL, стандартні індекси ігнорувалися. Я запакував фінальний запит у **Materialized View** (Матеріалізоване представлення). Це дозволило перенести все навантаження на нічний час (команда `REFRESH` виконується всього за 3 хвилини).

### 4. Result (Результат)

* **Стиснення даних:** Об'єм даних, що передаються в Power BI, скоротився в сотні разів за рахунок агрегації на рівні бази (за годинами та кварталами). Із 40+ мільйонів рядків данних отримав 300 тисяч вже прорахованих данних зі статистикою по координатам для побудови карти у візуалізації.
* **Швидкість для користувача:** Час завантаження звіту та реакція карти на кліки фільтрів зменшились з хвилин до **мілісекунд (0.05 сек)**, оскільки Power BI зчитує вже прорахований злімок.
* **Бізнес-цінність:** Директор отримав стильний (у брендових жовто-графітових кольорах), інформативний інструмент, який чітко показує реальний середній чек ($17.89), чайові ($1,93) та теплову карту Мангеттена без затримок.
